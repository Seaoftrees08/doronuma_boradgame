# doronuma_game_system.md

## 1. 全体構成と使用技術
インターネット上の画面と、裏側の審判役を分担させる構成とする。

* **画面の作成 (TypeScript + Next.js):** プレイヤーが見る画面、ボタン、カードの絵柄などを作るための道具。
* **ウェブサイトの置き場所 (Firebase Hosting):** 完成した画面をインターネット上に置き、アクセスできるようにする場所。
* **データの共有 (Cloud Firestore):** 「誰がカードを使ったか」「点数がどう変わったか」を全員の画面へ即座に伝えるリアルタイムの共有掲示板。
* **裏側の審判役 (Cloud Run):** 点数計算やカードを引く処理など、プレイヤー側で行うとズルができる作業を安全に行う仕組み。
* **プレイヤーの識別 (Firebase 匿名認証):** パスワード登録などをせず、ウェブサイトを開いた瞬間に見えない「参加券」を配り、画面を閉じても同じ人で復帰できるようにする仕組み。
* **自動更新 (Cloud Build + GitHub):** 新しい機能を追加した際に、自動で検査を行い、インターネット上のゲームを最新状態に書き換える仕組み。
* **時間管理システム（未来の予定表）:** 裏側の審判役の中に組み込まれる、自動で時間を管理する仕組み。

## 2. ロビー機能とゲーム開始の仕組み
あらかじめ用意した画面の枠組みの中に、「共有掲示板」から最新の設定データを読み込んで表示する。

* **部屋の作成と合流:**
    1.  主催者（ホスト）が部屋を作ると「合言葉」が発行される。
    2.  他の人は合言葉を入力して同じ待合室（ロビー）に入る。
* **参加判定（Ready機能）:**
    * ゲームに参加するプレイヤーは、ロビー画面で「Ready」にチェックを入れる。
    * チェックを入れていない人、またはゲーム開始後に新しく「ルームID」を入力して入ってきた人は、操作ができない「オブザーバー（見学者）」として参加する。
* **ルールの自由設定:**
    主催者はロビー画面で募集人数、山札の内訳、突然死カードを混ぜる範囲、1ターンの制限時間、割り込みの選択時間を変更できる。変更は即座に全員の画面に反映される。
* **ゲーム開始:**
    人数が揃い、参加者がReadyになった後、主催者が「開始」ボタンを押す。その合図が掲示板に書き込まれると全員の画面が切り替わり、裏側の審判役が山札の準備を始める。

## 3. 自動進行と通信管理
* **時間管理（自動予約システム）:**
    * 誰かの番が始まった瞬間、裏側の審判役は「未来の予定表」に「〇秒後にこのプレイヤーの時間切れ強制処理を実行する」という予約を書き込む。
    * 時間内にプレイヤーが行動した場合は、この予約を消去する。
    * 操作がないまま時間が経過すると、予定表が自動で審判役に命令を出し、強制的に「カードを2枚引き、上限に合わせてランダムに捨てる」処理を実行する。画面側の時計や通信状態に依存せず、確実に時間が管理される。
* **割り込みの処理:**
    攻撃が宣言されるとシステムが一時停止状態になり、カウントダウンが始まる。誰かが割り込むたびに時間は最初から数え直しになる。時間がゼロになった時点で、裏側の審判役が最後に出されたカードから順に効果を計算し、確定した結果だけを画面に送る。
* **通信の切断と見学者化:**
    * 2回連続で時間切れになったプレイヤーは「空席」扱いとなり、点数計算時に強制的に最下位（負け）となる。空席の人の番は、時間を待たずに一瞬で自動処理が行われ、順番が飛ばされる。
    * 通信トラブルなどで画面を閉じてしまい、敗北が確定した後にゲームに復帰したプレイヤーは、「オブザーバー（見学者）」となる。画面を見るだけで操作はできない。ゲーム終了後にロビーに戻れば、再びプレイヤーとして登録・参加できる。
* **主催者の切断:**
    主催者の通信が切断された、または空席扱いになった場合、残りの参加者からランダムに新しい主催者が選ばれ、操作の権限が移る。
